What are Microservices?

Published: July 29, 2025

Monolith: The Big Giant

The monolith. A single application, tightly packed, all features bundled together. One codebase. One deploy. Easy to begin with.

But once it grows? It turns into a bug infested machine where fixing something in the payment system could accidentally break the login. Slow builds, large deploys and huge effort

Enter Microservices

Microservices are like breaking a LEGO castle into small pieces. Each service is independent, has a clear function and focuses on a single responsibility

Instead of one app, we now have many tiny apps — maybe a user-service, a payment-service, and a notification-service. They talk to each other over APIs (like REST or gRPC)

If something goes wrong in one service, the rest of the system doesn’t necessarily collapse. So it's easy to debug and iterate

Why I’m Doing Microservices

I'm currently working on a web backend powered by Kubernetes. I wanted to learn how to:

  • Build scalable backend architectures
  • Use Docker and Kubernetes in real projects
  • Deploy services independently and test them in isolation

Microservices enables devs to have a playground to explore real-world backend problems while building something I can deploy and scale.

Challenges & Tradeoffs

But there are also cons to the pros. Microservices mean:

  • More moving parts
  • Harder debugging (logs are everywhere!)
  • DevOps complexity (CI/CD, monitoring, network issues)

But for large-scale systems or teams, the benefits start to outweigh the costs.

Conclusion

Microservices is not some fancy tech. It's thoughtful approach to scaling teams and software but at a cost of compexity

For my current learning journey, it makes perfect sense. They force me to think about real-world problems: APIs, failover, deployment, and system design.
